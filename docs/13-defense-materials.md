# 答辩与材料口径

## 项目介绍口径

妮妮 Kitchen Agent 是面向家庭厨房场景的一顿饭任务型 AI 终端。它不是菜谱推荐器，也不是普通聊天助手，而是通过记忆、库存、视觉和状态机，持续管理用户当前烹饪任务。

## 评委可能追问

### 1. 和普通菜谱推荐有什么区别？

普通菜谱推荐停在“给一道菜”。本项目覆盖规划、视觉修正、烹饪执行、状态控制、记忆更新和复盘。尤其是烹饪中途会根据实际食材和新偏好即时调整步骤。

### 2. 为什么做成终端，不做 App？

厨房场景中用户手湿、注意力分散，不适合频繁操作手机。终端强调大字号、单步动作、语音/按钮控制、状态常驻，更适合烹饪执行。

### 3. 视觉如果用 mock，算不算真实能力？

演示默认 mock 是为了线上答辩可复现。代码保留 real provider 接口。系统设计上，vision 输出被统一成结构化 observation，后续库存修正和菜谱调整是真实流程。

### 4. Agent 是否真的调用工具？

工具调用由后端 runtime 执行，memory、inventory、tool_events、terminal_state 会写入 SQLite。前端只渲染后端状态，不写死业务结果。

### 5. 为什么不用大模型处理下一步、暂停、继续？

这些是高频、确定性、低延迟指令，用状态机更稳定，也避免模型延迟和误判。复杂规划和语义调整才交给 Agent。

### 6. 菜谱能力是否泛化？

当前省赛版本不追求大规模菜谱库，先验证厨房 Agent 的任务闭环。recipe 模块使用可解释模板和调整规则，后续可以接入更大菜谱库和 RAG。

### 7. 长期记忆如何避免乱写？

只写入明确偏好、忌口和家庭约束，并通过 schema 存储为结构化 memory。演示中“不喜欢太酸”会影响当前步骤和下次建议。

### 8. 库存扣减是否准确？

demo 中以菜谱用量和视觉修正后的库存为基础做近似扣减，目标是验证流程闭环，不声称达到商用品质的精确称重。

### 9. 模型输出不稳定怎么办？

后端有 schema 校验、字段默认值、失败回退和 mock provider。线上演示优先保证可复现。

### 10. 项目主要技术贡献是什么？

不是单个模型能力，而是面向厨房任务的 Agent 编排：

- 结构化协议控制 UI。
- 状态机处理实时指令。
- 记忆、库存、视觉共同驱动烹饪流程。
- 后端状态和工具事件真实落库。

## 避免口径

不要说：

- 完全自主智能体。
- 真正理解厨房环境。
- 强实时多模态具身智能。
- 适配所有菜谱。
- 已经可商用落地。
- 全自动烹饪。
- 视觉识别完全准确。
- 硬件终端已实现。
- 模型自主调用任意工具。

推荐说法：

- 任务型 Agent 编排。
- 基于视觉 observation 更新库存和计划。
- 支持视觉输入和状态机即时控制。
- 当前验证一条闭环，菜谱模块可扩展。
- 面向厨房终端场景的原型系统。
- 辅助用户完成烹饪任务。
- 结构化记录明确偏好和约束。
- 网页模拟厨房终端交互形态。
- 模型输出受控 JSON，后端按白名单执行工具。
